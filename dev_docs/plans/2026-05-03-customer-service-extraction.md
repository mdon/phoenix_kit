# Customer Service module extraction

**Goal.** Extract `PhoenixKit.Modules.CustomerService` from PhoenixKit core
into a standalone hex package `phoenix_kit_customer_service`, mirroring the
prior `phoenix_kit_crm` extraction.

**Date.** 2026-05-03
**Branches.**
- core: new branch off `dev` in `/app`
- package: new repo at `/root/projects/phoenix_kit_customer_service`,
  initial commit on `dev` (target of upstream `BeamLabEU/phoenix_kit_customer_service`)

---

## Context (frozen вЂ” see researcher reports for evidence)

CS today lives entirely in:
1. `/app/lib/modules/customer_service/` вЂ” 22 files (1 context, 4 schemas, 1 events, 16 web)
2. `/app/lib/phoenix_kit_web/routes/customer_service.ex` вЂ” admin route module
3. `/app/lib/phoenix_kit_web/integration.ex` вЂ” three inline `live` blocks for
   user-dashboard routes (lines 583вЂ“596, 643вЂ“656) plus alias + safe_route_call
   wiring (lines 112, 386вЂ“397, 1158, 1191)
4. `/app/lib/phoenix_kit/module_registry.ex:411` вЂ” internal modules entry

**External deps used:** `PhoenixKit.{Module, Settings, RepoHelper, Utils.Date,
PubSub.Manager}`, `PhoenixKit.Users.{Auth.User, Roles}`, `PhoenixKit.Modules.Storage{,.File}`,
`PhoenixKit.Dashboard.{Tab, Registry}`, `PhoenixKit.Utils.Routes`,
`PhoenixKitWeb.{Components.LayoutWrapper, Components.Core.*, Gettext}`. All
satisfied by the package depending on `{:phoenix_kit, "~> 1.7"}`.

**Migrations stay in core.** Per `MODULE_EXTRACTION_GUIDE.md` В§5: "Don't put
migrations in the package." V35/V51/V53/V58/V72/V74/V75/V77 keep their
CS-related DDL in core. The `phoenix_kit_tickets`, `phoenix_kit_ticket_comments`,
`phoenix_kit_ticket_attachments`, `phoenix_kit_ticket_status_history` tables
and the `SupportAgent` system role remain core-managed. **No new migration is
generated by this extraction.**

**No `migrate_legacy/0` needed.** That callback is only for pre-uuid Integrations
data (CLAUDE.md). CS has no Integrations footprint. The settings rename
`tickets_*` в†’ `customer_service_*` is already done by V77.

**Gettext stays in core.** All ~125 CS strings remain in core's
`priv/gettext/*/LC_MESSAGES/default.po`. The package's LiveViews `use
PhoenixKitWeb, :live_view` which injects `gettext/1` from `PhoenixKitWeb.Gettext`
вЂ” it Just Works because the package depends on `phoenix_kit`.

**No tests today.** CS ships zero tests. We add a smoke test mirroring CRM
(behaviour compliance + tab metadata). Full coverage is out of scope.

**Two known fixes folded into extraction:**
- `enabled?/0` gets a `rescue _ -> false` clause (currently missing in core's
  internal CS, present in CRM/hello_world canon).
- `refresh_dashboard_tabs/0` keeps its existing `Code.ensure_loaded?` guard;
  do not remove it.

## Implementation slices

The work is partitioned so that 3вЂ“4 implementers can run in parallel without
stepping on each other. Each slice has its own working directory, no overlap.

### Slice A вЂ” package skeleton & test scaffold (no code logic)
Working dir: `/root/projects/phoenix_kit_customer_service` (create empty).
1. `mix.exs` cloned from CRM, renamed to `:phoenix_kit_customer_service` /
   `PhoenixKitCustomerService.MixProject`. **Commit a Hex-clean dep**
   (`{:phoenix_kit, "~> 1.7"}`); after first PR push, switch to `path: "/app",
   override: true` locally without committing (per CLAUDE.local.md).
2. `.formatter.exs`, `.gitignore`, `LICENSE`, `README.md`, `CHANGELOG.md` cloned/adapted from CRM.
3. `config/config.exs`, `config/test.exs` set up `PhoenixKitCustomerService.Test.Repo`,
   DB name `phoenix_kit_customer_service_test`.
4. `test/support/test_repo.ex`, `test/support/data_case.ex`, `test/test_helper.exs`
   modeled on `phoenix_kit_hello_world/test/`.

   PhoenixKitCustomerService do end` so `mix compile` works for slice B handoff.
**Done when:** `cd /root/projects/phoenix_kit_customer_service && mix deps.get
&& mix compile` succeeds.

## Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| `safe_route_call` removal in slice F breaks admin routing | low | core auto-discovers via `compile_external_admin_routes` (integration.ex:958-996); CRM works this way. Ecommerce still uses explicit `safe_route_call`, but auto-discovery is preferred for new extractions. |
| Gettext strings stop resolving at runtime | low | package's `use PhoenixKitWeb, :live_view` injects core's Gettext backend; same as CRM today |
| Storage/MediaBrowser couplings break across module boundary | medium | Storage stays in core; package depends on `phoenix_kit`. Verify in slice F end-to-end test. |
| `SupportAgent` role disappears when re-running migrations | low | role stays in core's V35; nothing about extraction changes V35 |
| Hex constraint mismatch (`phoenix_kit ~> 1.7`) when core is `1.7.104` | low | `~> 1.7` admits any `1.7.x`; package on Hex never published anyway during local dev |
| Two `customer_service` route_module entries (core + package) cause duplicate routes | medium | slice F removes core's route module before package adds its own; verify with `mix phx.routes` |
| Inlined user-dashboard routes in core silently shadow package's | medium | slice F replaces inline blocks with `Code.ensure_loaded?` guards вЂ” only one definition wins |
