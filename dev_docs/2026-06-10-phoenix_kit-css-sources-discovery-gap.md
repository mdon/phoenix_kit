# Handoff: CSS `@source` auto-discovery silently produces an empty `_phoenix_kit_sources.css`

**Date:** 2026-06-10
**Audience:** the agent working in the **`phoenix_kit` (core)** package
**Status:** bug report + proposed fix for implementation
**Severity:** medium — silent, hard-to-diagnose UI breakage in any host that uses an
external PhoenixKit module (e.g. `phoenix_kit_ai`) with Tailwind v4.
**Reported from:** `phoenix_kit_ai` side, after a live host (`langust`) showed a broken
admin table. The defect itself is entirely in **core**, not in `phoenix_kit_ai`.

All paths below are in the **`phoenix_kit` repo** (`lib/...`). They appear in a consuming
app as `deps/phoenix_kit/lib/...`. Line numbers are from `phoenix_kit` **1.7.138** (the
version vendored into `phoenix_kit_ai`'s deps at time of writing) — confirm against the
version the affected host actually runs as step 0.

---

## 1. Symptom (what the user saw)

On a host app that has `phoenix_kit` + `phoenix_kit_ai` installed, the **AI Usage** admin
page (`phoenix_kit_ai`'s `endpoints.html.heex`, the "Recent Requests" table) renders only
the columns that have **no responsive Tailwind variant**:

```
Time | Endpoint | Status | Actions
```

Every column gated behind a responsive variant — `Price` (`hidden sm:table-cell`),
`Tokens` / `Latency` (`hidden md:table-cell`), `Model` / `Prompt` / `Source`
(`hidden xl:table-cell`) — is **invisible at every viewport width**, even at 1751px (well
above the `xl` 1280px breakpoint). The `<th>`/`<td>` are present in the DOM with their
classes; they just never get a `display: table-cell` rule.

This is the tell-tale signature of **"the responsive variant utilities were never compiled
into the CSS bundle."** `hidden` is generated (the host uses it elsewhere); `md:table-cell`,
`sm:table-cell`, `xl:table-cell` are used **only inside the `phoenix_kit_ai` dependency's
templates**, so if Tailwind never scans that dependency, those utilities are absent and
`hidden` is never overridden.

### Why it's fatal under Tailwind v4

The affected host's `assets/css/app.css` begins with:

```css
@import "tailwindcss" source(none);
```

`source(none)` **disables Tailwind v4 automatic content detection** — only the explicit
`@source` directives are scanned. So the *entire* mechanism for getting a dependency's
classes compiled is the generated `@source` list. If that list is missing
`deps/phoenix_kit_ai`, the dep is invisible to Tailwind. (Under Tailwind v3 / auto-content
this bug may be masked, which is probably why it went unnoticed.)

### The smoking gun

On the affected host:

```
$ cat assets/css/_phoenix_kit_sources.css
/* Auto-generated by PhoenixKit — do not edit manually.
   Regenerated on each compilation from css_sources/0 callbacks. */
```

The file is **present, freshly generated, and empty of `@source` lines.** `app.css`
correctly does `@import "./_phoenix_kit_sources.css";`. So the wiring is all in place — the
generator just produced zero sources.

---

## 2. How the mechanism is supposed to work

Three actors, all in core except the marker:

1. **`mix phoenix_kit.update` / installer** (`lib/mix/tasks/phoenix_kit.update.ex`,
   `lib/phoenix_kit/install/css_integration.ex`): one-time wiring — registers the
   `:phoenix_kit_css_sources` compiler in the host `mix.exs`, writes the `app.css`
   integration block (`@source "../../deps/phoenix_kit";` + `@import
   "./_phoenix_kit_sources.css";`), rebuilds assets. **Does not** write per-module
   `@source` lines itself.
2. **`:phoenix_kit_css_sources` compiler** (`lib/mix/tasks/compile.phoenix_kit_css_sources.ex`):
   runs every `mix compile`. Calls `ModuleDiscovery.discover_external_modules/0`, then for
   each module calls `css_sources/0` and writes the resolved `@source` lines into
   `assets/css/_phoenix_kit_sources.css`. **This is the part that failed** — it emitted an
   empty list.
3. **`use PhoenixKit.Module`** in the external module: persists `@phoenix_kit_module true`
   in the beam so discovery can find it. `phoenix_kit_ai` does this correctly
   (`PhoenixKitAI` at `lib/phoenix_kit_ai.ex:79`, and its `css_sources/0` returns
   `[:phoenix_kit_ai]`).

So the host did nothing wrong: the compiler is registered (the file exists and regenerates),
`app.css` imports it, and the external module is marked. **Discovery returned nothing.**

---

## 3. Root cause: discovery diverges from its own documented contract

`lib/phoenix_kit/module_discovery.ex` moduledoc promises:

> Uses the same pattern as Elixir's protocol consolidation: scans `.beam` files for
> persisted `@phoenix_kit_module` attributes via `:beam_lib.chunks/2`. **No module loading
> required — pure file I/O.** … scans only deps that depend on `:phoenix_kit`.

But the implementation does **not** do filesystem scanning of deps. It gates everything on
the **runtime application controller state**:

```elixir
# module_discovery.ex
defp phoenix_kit_dependent_apps do
  for {app, _, _} <- :application.loaded_applications(),   # <-- runtime app list
      app != :phoenix_kit,
      depends_on_phoenix_kit?(app) do
    app
  end
end

defp depends_on_phoenix_kit?(app) do
  case :application.get_key(app, :applications) do          # <-- needs app loaded
    {:ok, apps} -> :phoenix_kit in apps
    _ -> false
  end
end
```

`scan_app_ebin/1` (the only actual filesystem path) is reachable **only for apps already
returned by `phoenix_kit_dependent_apps/0`** — i.e. apps already loaded. If an app never
makes it into `:application.loaded_applications()`, it's invisible. There is no
deps-directory walk anywhere, despite the moduledoc.

The compiler tries to paper over this (`compile.phoenix_kit_css_sources.ex:run/0`):

```elixir
for dep <- Mix.Dep.cached() do
  Application.ensure_loaded(dep.app)        # best-effort; return value ignored
end
```

This is fragile and insufficient in practice. Known ways it still yields an empty list:

- `Application.ensure_loaded/1` returns `{:error, _}` silently when the dep's `.app` isn't
  available at that compile phase / first cold build; the app then isn't in
  `loaded_applications()` for discovery and the failure is swallowed.
- Compiler ordering / cold-build first pass: the generated file gets written **empty**, and
  because the compiler only rewrites **when the content changes** (`if existing != content`),
  an empty file written once can persist across later builds even after the app *would*
  load — the content is "empty" both times, so it's never rewritten with the real sources.
- Older `phoenix_kit` versions on the host may predate even the `ensure_loaded` loop.

I verified that `phoenix_kit_ai`'s `.app` **does** list `:phoenix_kit` in `applications`
(via `extra_applications: [:phoenix_kit]`), so `depends_on_phoenix_kit?/1` would pass *if*
the app were loaded — which confirms the failure is upstream of that check: the app simply
isn't in `loaded_applications()` when discovery runs on the affected host.

### Net

The whole point of "scan beam files, no loading required" is to be deterministic at compile
time. The implementation instead depends on runtime load state, so discovery is
nondeterministic and, when it loses, fails **silently** with an empty file that looks
correct.

## 4. Reproduction

1. Fresh Phoenix app, Tailwind v4, `app.css` with `@import "tailwindcss" source(none);`.
2. Add `phoenix_kit` + `phoenix_kit_ai`; run the installer so the compiler + `app.css`
   integration block are wired.
3. Cold build: `rm -rf _build && mix compile`.
4. Inspect `assets/css/_phoenix_kit_sources.css` — observe it is empty (no
   `@source ".../deps/phoenix_kit_ai";`).
5. `mix assets.deploy`; load the AI Usage page wide — Tokens/Price/etc. columns missing.

(If it doesn't repro on a warm build, that's consistent with the load-timing
nondeterminism — the cold/first-build path is the most reliable repro.)

---

## 5. Proposed fix (make discovery match its contract)

**Primary:** rewrite `ModuleDiscovery.scan_beam_files/0` to do real filesystem scanning of
**dependency ebin directories**, independent of `:application.loaded_applications()`:

- Enumerate dep apps from the build, not the app controller — e.g. each
  `Mix.Dep.cached/0` entry, or directly glob `_build/#{Mix.env()}/lib/*/ebin`.
- For each candidate app, read `<app>.app` from `ebin` to check whether `:phoenix_kit` is in
  its `applications` (filesystem read, no load), keeping the "only phoenix_kit-dependent
  deps" optimization the moduledoc describes.
- For matching apps, glob `ebin/*.beam` and use `:beam_lib.chunks(path, [:attributes])` to
  keep those with `@phoenix_kit_module == [true]` (the existing `phoenix_kit_beam_file?/1`
  helper already does exactly this — reuse it).
- Keep `Application.get_env(:phoenix_kit, :modules, [])` as an additional explicit fallback,
  and keep the runtime/loaded-app scan as a *supplementary* source for the live
  `ModuleRegistry` runtime path. Merge + dedupe.

This must work in a **compile-time** Mix task context (no running OTP app tree). The result:
discovery returns the same set whether or not apps happen to be loaded.

**Secondary hardening (cheap, high value):**

1. In `compile.phoenix_kit_css_sources.ex`, **warn loudly when discovery yields zero
   sources but phoenix_kit-dependent deps exist** on disk:
   ```
   [PhoenixKit] _phoenix_kit_sources.css generated with 0 sources, but found
   phoenix_kit-dependent deps: [:phoenix_kit_ai]. Their Tailwind classes will NOT be
   compiled. See <doc link>.
   ```
   A silent empty file is the worst part of this bug. Even without the primary fix, this
   turns a 2-hour CSS mystery into a one-line build warning.
2. Don't let an **empty** result overwrite/persist a previously non-empty file without at
   least the warning above (guards the "written empty once, never rewritten" trap).
3. Check the return value of `Application.ensure_loaded/1` and log at debug when a dep can't
   be loaded, so the failure is observable.

**Optional, defense-in-depth for the consumer side:** `phoenix_kit_ai`'s `css_sources/0`
could also return a path-dep absolute fallback (the pattern documented in
`PhoenixKit.Module` — `[:phoenix_kit_ai, @source_root]`). That does **not** fix this bug
(the module is never discovered, so `css_sources/0` is never called), but it's the
documented belt-and-suspenders for path-dep installs and worth noting. The real fix is in
core discovery.

## 7. Tests

- **ModuleDiscovery (compile-time, no running apps):** in a fixture that simulates a deps
  ebin layout (a `.app` listing `:phoenix_kit` + a `.beam` carrying
  `@phoenix_kit_module true`), assert `discover_external_modules/0` finds the module
  **without** that app being in `:application.loaded_applications()`. This is the
  regression lock — it must pass with the app unloaded.
- Negative: an app whose `.app` does **not** depend on `:phoenix_kit` is excluded; a beam
  without the attribute is excluded.
- Config fallback still merges and dedupes.
- **Compiler:** with one external module present, `_phoenix_kit_sources.css` contains its
  `@source` line; with none, the file is empty **and** a warning is emitted when a
  phoenix_kit-dependent dep is present.

---

## 8. Acceptance criteria

1. On a cold build (`rm -rf _build && mix compile`) of a host with `phoenix_kit_ai`,
   `assets/css/_phoenix_kit_sources.css` contains `@source "../../deps/phoenix_kit_ai";`
   with **no** `config :phoenix_kit, modules: [...]` entry required.
2. The AI Usage table shows Tokens/Price/Model/etc. at appropriate widths after
   `mix assets.deploy`, with Tailwind v4 `source(none)`.
3. If discovery ever returns zero sources while phoenix_kit-dependent deps exist, the build
   prints a clear warning (no silent empty file).
4. Existing runtime `ModuleRegistry`/route discovery behavior is unchanged.

## 9. Interim workaround (already given to the affected host)

Until the core fix ships, hosts can use the supported config fallback (deterministic, not a
manual CSS hack):

```elixir
# config/config.exs
config :phoenix_kit, modules: [PhoenixKitAI]
```

`discover_external_modules/0` merges `Application.get_env(:phoenix_kit, :modules, [])`, so
this forces the `@source` to be emitted regardless of load timing. Then re-run
`mix phoenix_kit.update` (or `mix compile && mix assets.deploy`). A manual
`@source "../../deps/phoenix_kit_ai";` line in `app.css` also works but shouldn't be the
permanent answer.
